Chapter One: Introduction to Software Engineering Tools and Practices

Course Outcomes - why you are taking this course?

  • Become familiar with the practices used by Software Engineers for creating software applications.
  • Become familiar with the various modern tools used by Software Engineers for creating applications.
  • So that you can get a job and work in a team.

What is the Software Life Cycle?

The stages of developing a software application:

Flow of Software Life Cycle: [Requirements] | v [Design] | v [Implementation] | v [Verification] | v [Maintenance]

Software practices vs. Process

Software Process: Defines the steps you take to develop good software. E.g., SDLC.

Software Practices: Specific things you do as part of the process. That is, practices are activities that implement the process.

Software development practices that might take place in each phase:

Phase Practices
Requirements Stakeholder meeting and interviews, Use case modeling, SRS
Design Modeling with UML models (class, sequence, activity diagrams), prototyping
Implementation Convert design into working code, code review, pair programming, version control...
Verification Verifying that the software works correctly (unit testing, integration, system testing)
Maintenance Version updates, user feedback analysis, improving existing code

What are Software tools?

  • Software Engineering Methods are intended to make software engineering more systematic.
  • The computer based tools are Software development environments that are intended to assist the software life cycle processes.
  • Tools are often designed to support particular software engineering methods.
  • Are programs supports different stages of software development.

Software Engineering Tools and Methods

  • Software Engineering tools represent:
    • A set of management and technical tools to support software development.
    • Usually integrated in a coherent framework.
  • Methods: usually provide techniques and procedures for performing identifiable tasks, and guidelines for checking both the process and the product.

Software Engineering Tools Categories

  • Software Requirement Tools
  • Software Design Tools
  • Software Construction Tools
  • Software Testing Tools
  • Software Maintenance Tools
  • Software Configuration Management Tools

Software Requirements Tools

These Tools are classified into two categories:

  1. Requirements modeling tools: These tools are used for clarifying, visualizing and analyzing user and system requirements in structured diagram or models. Eg. StarUML (use case, activity, ER models).
  2. Requirement traceability tools: It helps to track and manage the life of requirements through SDLC. Maintain links between requirements, and their corresponding design, code, and testing cases.

Chapter Two: Requirements Analysis via Use Cases

What is Requirements Analysis?

It is the process of examining, understanding, and refining user's needs and system requirements to ensure they are clear, complete, and feasible before development begins.

We perform Requirements Analysis:

  • To ensure that software requirements are clear, correct, and testable.
  • Help the customer/client determine what the cost will be.
  • Produce a solution that best satisfies the given problem.

"Be aware of the guy in the room" - Steve McConnell

Requirement

  • A Requirement is a systematic description of the functions, features, and constraints of a software system.
  • It is a description of what a system should do or how it should perform.
  • When all Requirements are met, your application is complete!
  • Software Quality is determined in part as successful conformance to Requirements.

User Stories

User Stories are a common way of expressing a Requirement of what the system must do.

User Story Format: "As a [specific type of user], I would like to [achieve some goal] [in order to [justification]]"

Examples:

  • As a Student, I would like to view open courses in order to create a schedule.
  • As a Faculty, I would like to view the students registered for the course sections I am teaching.
  • As the Registrar, I would like to create courses and course sections.

A complete set of User Stories describe everything a system can do. They are however frequently incomplete at the start.

UML Review: What is UML and Why we use UML?

UML = Unified Modeling Language.

  • Modeling: Describing a software system at a high level of abstraction. Help acquire an overall view of a system.
  • UML is not dependent on any one language or technology.

UML History

  • 2003: UML 2.0
  • 2001: UML 1.4
  • 1999: UML 1.3
  • 1997: UML 1.0, 1.1
  • Methods combined: Booch '91/'93, OMT-1/2, Unified Method 0.8.

Types of UML Diagrams

  • Use Case Diagram
  • Class Diagram
  • Sequence Diagram

UML diagram tools: StarUML, ArgoUML.

Use-Case Diagrams

Structure of Use Case Diagram: [Actor] --(line)-- (Use Case Oval) Boundaries: The system box containing Use Cases. Actors: Stick figures outside the boundary (e.g., Client, Employee, Supervisor). Use Cases: Ovals inside the boundary (e.g., Borrow, Order Title, Fine Remittance).

Relationships

1. Include

Used when a use case always uses another use case's behavior. The included use case is always executed as part of the base use case.

(Create Schedule) --- << include >> ---> (View Open Courses)

Every time a student creates a schedule, the system must include the "View Open Courses" functionality.

2. Extend

A dotted line labeled <<extend>> with an arrow toward the base case. The extension happens only under certain conditions.

(Check for Schedule Conflicts) --- << extend >> ---> (Create Schedule)

Elements of Use Case Diagrams

  • Actor: An external agent that interacts with the system.
    • Actors for some systems are just conventional "users".
    • Can be different types (Student, Faculty, Registrar).
    • An Actor does not have to be a person; can be external systems.
  • Goal: The specific thing accomplished as a result of executing a Use Case.
  • Use Case: An effective technique for narrating how a system works in fulfilling a User Story/Requirement.
    • Use case names are usually short verb + noun phrases (e.g., "Create Account").

Scenarios

A Scenario is a narrative that describes what happens within a specific Use Case.

  • Main Path (Sunny Day): Describes what happens in the "normal" case. The Goal is always achieved.
  • Alternate Scenarios (Rainy Day): Describe atypical or exceptional situations. Can have different steps, additional steps, or recovery steps.

"Plan for things to go wrong!"

Boundaries, Preconditions, Postconditions

  • Boundaries: Every Use Case must have a definite Starting and Stopping point.
  • Precondition: Conditions that must exist before a Use Case can proceed (e.g., must be logged in).
  • Postcondition: Artifacts created or state changes after achieving a goal (e.g., database updated, receipt generated).

Requirement Analysis via Requirement Management Tool (ReqView)

Requirements management tools (RM tools) help remove ambiguity, assumptions, and wishful thinking. The goal is clear, realistic, and agreed-upon requirements.

Basic vs. Dedicated Tools

  • Basic: Spreadsheets (Excel) or Word docs. Problematic because they rely on manual updates, version control is hard, and managing feedback is difficult.
  • Dedicated (e.g., ReqView): Help track, test, analyze, visualize, and communicate.

ReqView Features

  • Capture structured requirements.
  • Track traceability to design, tests, and risks.
  • User Defined Attributes.
  • History Tracking.
  • Export to HTML, Word, CSV.
  • Import from Word and CSV.
  • Offline Collaboration.

1. High Level Design (HLD)

Describes how the system is divided into modules, and how those modules interact. Refers to the overall design of a system.

Examples: Sequence diagrams (at a high level), Architecture diagrams, Component diagrams, Deployment diagrams.

HLDs: Use Case Textual Analysis

Textual Analysis of Use Case scenarios is used to create high-level designs. It is a quick way to make a "first guess" at the classes.

  • Nouns in Use Cases are candidates for Classes/Objects.
  • Verbs in Use Cases are usually the Methods the classes implement.

Object Classifications in Textual Analysis

  1. Boundary Objects: Model the system boundary (User Interface, screens, interfaces to external actors). Actors can only talk to boundary objects.
  2. Control Objects: Represents a manager that makes decisions (e.g., logic). In simple systems, usually the application itself. Controllers can talk to boundary and entity objects.
  3. Entity Objects: Data store or persistence element that captures information. Entity objects usually only talk to controllers and boundary objects.

High-Level Sequence Diagrams

Draw a UML HL Sequence Diagram for each Use Case showing interaction between objects.

  • Lifelines: Vertical dashed lines representing the object's life.
  • Creation: 'Create' message; object life starts.
  • Activation: Rectangular stripes on lifeline.
  • Deletion: 'X' on lifeline.

2. Detailed Design (LLD)

Focuses on the internal design of modules. Describes class structures, methods, relationships, algorithms, and detailed logic. Example: Class diagrams, State diagrams, Activity diagrams.

Identifying Classes

  1. List all objects from Sequence Diagrams (eliminate redundancies).
  2. List messages sent to each class (these become methods).
  3. Create a list of attributes based on context.

Class Design Principles

  • High Cohesion: Classes should do one thing well. Independent. Components of a module are directed towards a single task.
  • Low Coupling: Modules should be very lightly dependent on each other. Changes internal to one class shouldn't require changes to others.

UML Class Diagrams

Represents classes and their relationships. A rectangle subdivided into three compartments:

------------------------ | Account | <- Name ------------------------ | - balance: double | <- Attributes | + rate: double = 2.5 | ------------------------ | + deposit(): void | <- Operations | + withdraw(): void | ------------------------

Visibility Modifiers:

  • + Public
  • # Protected
  • - Private

Class Relationships

  • Generalization: Parent/child relationship (Inheritance). E.g., Regular Customer and Loyalty Customer inherit from Customer.
  • Association: Relationship between instances. E.g., Student enrolls in Course. Has role names and multiplicity (1, 0..1, *, 1..*).
  • Aggregation: "Whole/Part" relationship. Informal. Container and Containees have no special access privileges. If container is destroyed, parts may survive. (Empty diamond).
  • Composition: Stronger "Whole/Part". Every part belongs to only one whole. If whole is deleted, parts are deleted. (Filled diamond). E.g., Chessboard and Squares.

3. Source Code Generation

Generating source code from UML class diagrams. Attributes and methods in the diagram are converted to code stubs.

4. Synchronizing Source Code with UML Models

  • Forward Engineering: Model -> Code.
  • Reverse Engineering: Code -> Model.
  • Round-trip Engineering: Synchronizing both. If you change the model, code updates. If you edit code, model updates.
